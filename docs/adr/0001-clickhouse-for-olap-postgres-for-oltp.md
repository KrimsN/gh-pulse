# ADR 0001: ClickHouse для аналитики, PostgreSQL для метаданных

**Статус:** принято
**Дата:** 2026-07-15

## Контекст

GH Pulse отвечает на два очень разных типа вопросов: аналитические агрегаты поверх 100M+ событий
(«топ репозиториев по звёздам за час») и транзакционные обращения к метаданным («валиден ли этот
API-ключ, какие отчёты он сохранил»). Одно хранилище не обслужит оба хорошо.

## Решение

- **ClickHouse (OLAP)** держит поток событий и его агрегаты в materialized views. Колоночное
  хранение, векторизованное выполнение и MergeTree превращают агрегат по часовому окну поверх 100M
  строк в задачу на <100 мс. Это ядро проекта.
- **PostgreSQL (OLTP)** держит `api_keys` и `saved_reports`: небольшие, транзакционные, реляционные,
  часто обновляемые данные. Строчное хранение, настоящие транзакции, внешние ключи, миграции Alembic.

Это классический OLAP/OLTP-шов: тяжёлые агрегаты и мелкие транзакционные обращения имеют
принципиально разные профили нагрузки, и каждое хранилище настроено ровно под свой.

## Отвергнутые альтернативы

- **PostgreSQL для всего:** агрегаты поверх 100M строк потребовали бы тяжёлой предагрегации и всё
  равно проиграли бы ClickHouse на 1–2 порядка. Убивает заголовочное обещание проекта.
- **ClickHouse для всего:** плохо ложится на частые мелкие обновления и транзакционную целостность
  ключей/отчётов; нет нормальной семантики UPDATE/DELETE.

## Последствия

- Два датастора в эксплуатации — приемлемо, оба есть в docker-compose.
- Слой API джойнит данные между хранилищами в коде приложения (проверка ключа в PG, данные в CH).
  На этом масштабе нормально; задокументировано, чтобы не приняли за недосмотр.
