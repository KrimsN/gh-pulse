# ADR 0006: structlog для структурированного логирования

**Статус:** принято
**Дата:** 2026-07-15

## Контекст

`pulse-api` — сервис, где логи потребляются агрегатором как JSON-поток, а не читаются человеком в
терминале. Внутри одного запроса нужен сквозной контекст (`trace_id`, `path`, `method`), доступный
всем логам без явной передачи логгера по стеку вызовов. При этом сервис работает поверх стороннего
кода (uvicorn, asyncpg, clickhouse-connect), который логирует через стандартный `logging` — свой
логгер должен встраиваться в этот поток, а не подменять его.

## Решение

`structlog` как единственная библиотека логирования во всех Python-сервисах.

- **`structlog.stdlib.LoggerFactory`** отправляет записи structlog в стандартный `logging` — свой
  логгер встраивается в общий поток, а не подменяет его.
- **`structlog.stdlib.ProcessorFormatter`** — форматтер stdlib-хэндлера, который и приводит к общему
  виду чужие записи: `foreign_pre_chain` повторяет для них те же процессоры, поэтому лог uvicorn или
  asyncpg получает тот же timestamp, уровень и JSON-рендер, что и собственный. Отдельного
  перехватчика не нужно, но и бесплатно это не происходит — см. «Последствия».
- **`structlog.contextvars.bind_contextvars`** привязывает `trace_id`/`path`/`method` к контексту
  запроса один раз в `TraceIdMiddleware` — дальше любой `logger.info(...)` внутри обработки запроса
  несёт этот контекст автоматически.
- **`JSONRenderer`** как терминальный процессор — структурированный вывод является основным
  сценарием, а не опцией поверх текстового форматирования.

## Отвергнутые альтернативы

- **`loguru`:** лучше DX из коробки для скриптов и CLI — цвета, ротация файлов, читаемые трейсбеки
  без настройки. Но подменяет логгер целиком: чтобы перехватить stdlib-логи сторонних библиотек,
  нужен отдельный `InterceptHandler`. У него нет прямого аналога `contextvars`-биндинга для
  request-scoped контекста — эту связку пришлось бы собирать вручную поверх `loguru.configure`.
  Ради сервиса, где вывод всегда JSON, такая надстройка не окупается.
- **Стандартный `logging` без обёртки:** структурированный JSON и биндинг контекста пришлось бы
  реализовывать самостоятельно поверх `Filter`/`Formatter` — то, что structlog даёт готовым
  processor pipeline.

## Последствия

- Единая точка конфигурации логирования (`configure_logging` в `app/logging_config.py`) для любого
  нового Python-сервиса — `pulse-consumer` подключает её так же.
- Сторонний код, логирующий через stdlib `logging`, попадает в тот же JSON-поток — но только пока
  его логгеры доходят до root-хэндлера. uvicorn, например, конфигурирует логирование сам (`dictConfig`
  до импорта приложения) и выдаёт своим логгерам собственные хэндлеры с `propagate=False`; их
  приходится возвращать в общий поток явно. Цена решения: конфигурация знает о конкретных логгерах
  сервера, и при смене их имён или схемы вывод молча вернётся к тексту — это проверяется прогоном
  логов через парсер JSON, а не чтением глазами.
