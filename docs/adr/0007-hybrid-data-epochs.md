# ADR 0007: Гибридная стратегия данных — бэкфилл до октября 2025, live-поток из настоящего

**Статус:** принято
**Дата:** 2026-07-15

## Контекст

Перед проектированием схемы ClickHouse был осмотрен реальный час GH Archive. Данные разошлись с
ожиданиями настолько, что это влияет на продукт, а не только на типы колонок.

**Форма события изменилась в первой половине октября 2025.** `PushEvent.payload` потерял поля
`commits`, `size` и `distinct_size`:

| | до | после |
|---|---|---|
| `PushEvent.payload` | `before, commits, distinct_size, head, push_id, ref, repository_id, size` | `before, head, push_id, ref, repository_id` |

Граница локализована между `2025-10-01-15` (поля есть) и `2025-10-15-15` (полей нет).

**Одновременно упал объём событий, порождённых людьми.** Сравнение двух понедельников, 15:00 UTC:

| Метрика | `2024-06-03-15` | `2026-06-01-15` |
|---|---|---|
| Событий за час | 256 682 | 149 565 |
| Объём (gz / распакованный) | 140 МБ / 1.1 ГБ | 20 МБ / 97 МБ |
| **WatchEvent (звёзды)** | **10 006 (3.90%)** | **194 (0.13%)** |
| PullRequestEvent | 20 224 (7.88%) | 581 (0.39%) |
| ForkEvent | 2 855 (1.11%) | 44 (0.03%) |
| PushEvent | 137 936 (53.74%) | 126 758 (84.75%) |

`PushEvent` просел на 8%, тогда как звёзды — в 51 раз, форки — в 65 раз. Картина устойчива:
в `2026-07-14-15` звёзд 69 (0.04%), в `2026-06-15-18` — 139 (0.09%). Новых типов событий не
появилось, старые не исчезли — изменились объёмы.

Это ломает основной аналитический сценарий. «Топ репозиториев по звёздам за час» на потоке в
194 звезды на весь GitHub возвращает список, где у каждого репозитория одна-две звезды: результат
статистически неотличим от шума, и никакая оптимизация запроса этого не исправит.

## Решение

**Исторический бэкфилл берётся из архивов до октября 2025, live-контур — из текущего потока.**

- **Бэкфилл** (объём корпуса, аналитика, бенчмарки): архивы по 2025-09 включительно. Там есть
  звёзды в количестве, на котором trending осмыслен, и богатый payload.
- **Live** (контур реального времени): GitHub Events API в его сегодняшней форме.
- **Обе формы нормализуются в `gh-collector` к одной канонической схеме события.** Поля, которых
  в новой форме нет, не могут быть обязательными: коллектор проставляет им нулевое значение.

Каноническая схема события переживает обе эпохи без изменений — она и не опиралась на `commits`.
Единственное затронутое поле, `payload_size`, сохраняется (см. последствия).

## Отвергнутые альтернативы

- **Только свежие данные (2026).** Компактно: 100M событий ≈ 667 часов ≈ 13 ГБ gz против 55 ГБ для
  той же цифры на архивах 2024. Но trending по звёздам пришлось бы заменить на trending по
  push-активности — а это другой продукт, и заявленный сценарий исчезает.
- **Только исторические данные.** Убирает live-контур, ради которого в архитектуре есть Kafka и
  консьюмер. Проект превратился бы в batch-аналитику над статическим дампом.
- **Синтетические звёзды поверх реального потока.** Нарушает правило «только реальные данные»:
  измерения на выдуманном распределении ничего не говорят о системе.

## Последствия

- **Данные до и после октября 2025 несопоставимы по `payload_size`.** В старых данных размер
  payload у `PushEvent` отражал число коммитов (переменная величина); в новых это фактически
  константа — медиана 176 байт при разбросе 171–292. У `WatchEvent` payload равен ровно 20 байт
  всегда. Любой график `payload_size` во времени даст скачок на границе октября 2025, который
  выглядит как смена поведения пользователей, но является сменой формата. Поле остаётся грубым
  индикатором в пределах одной эпохи; метрики активности на нём не строятся.
- **Число коммитов в пуше недоступно для новых данных.** Метрика «объём пуша» может быть посчитана
  только на историческом корпусе.
- **`gh-collector` обязан уметь обе формы payload с первого дня** и покрывать обе тестами — иначе
  live-путь сломается на полях, которых в нём нет.
- **Свежесть корпуса ограничена.** Основной корпус заканчивается сентябрём 2025; данные между ним и
  live-контуром содержат мало звёзд. Эндпоинты, чувствительные к звёздам, работают по историческому
  диапазону — это отражается в ответе `/api/v1/stats` (свежесть данных).
- Все цифры выше получены на реальных файлах GH Archive, ни одна не оценочная. Воспроизводятся
  прямым подсчётом по скачанному часу — например, распределение типов:

  ```bash
  curl -fsSL https://data.gharchive.org/2026-06-01-15.json.gz | gunzip -c \
    | jq -r '.type' | sort | uniq -c | sort -rn
  ```
