# ADR 0009: OpenTelemetry-трассировка через Kafka-заголовки, Jaeger как backend

**Статус:** принято
**Дата:** 2026-07-20

## Контекст

Пайплайн проходит через три независимых процесса (`gh-collector` → Kafka → `pulse-consumer` →
ClickHouse, отдельно `pulse-api` поверх тех же данных), каждый сервис уже пишет структурные логи
(`structlog`, ADR 0006) и метрики Prometheus, но между процессами ничего не связывает одну запись
лога в `gh-collector` с записью того же события в `pulse-consumer` — только совпадение `event_id` в
тексте лога, которое приходится искать руками. Нужен способ увидеть один event сквозь границу
процессов и стадию (fetch/produce → consume → insert), не разбирая логи трёх сервисов вручную.

Дополнительная сложность: `pulse-consumer` обрабатывает события батчами (`getmany()` отдаёт до
20 000 записей разом, `consumer.consumer.run`), а не по одной. Значит, «один trace_id на всю жизнь
события» в буквальном смысле «одно дерево span'ов с одним родителем» не работает на границе
consume→insert: у одного `INSERT` в ClickHouse одновременно может быть до 20 000 родителей, а у
span'а — ровно один.

## Решение

**OpenTelemetry Tracing SDK** во всех трёх сервисах, экспорт по OTLP/gRPC в **Jaeger**
(`jaegertracing/all-in-one`, OTLP включён нативно — свой отдельный OTel Collector не нужен).

- **`gh-collector`:** каждый вызов `producer.Producer.Produce` — свой root span
  (`kafka.produce`, semantic conventions `messaging.system=kafka`, `messaging.destination=<topic>`).
  W3C `traceparent` инжектится в заголовки Kafka-записи (`kgo.RecordHeader`) через
  `propagation.TraceContext` — стандартный, а не самодельный формат: любой Python/Go/Java-агент
  извлекает его без специального кода на другой стороне.
- **`pulse-consumer`:** на каждый непустой батч из `getmany()` — один span
  `consumer.process_batch` (`messaging.batch.message_count` в атрибутах). Для каждого сообщения
  батча `traceparent` из его заголовков превращается в **`trace.Link`** на span батча — не в
  родителя. Это ровно то, что рекомендуют семантические конвенции OTel для batch receive: у span'а
  один родитель, но сколько угодно links, и каждый link — это честная ссылка «этот батч включает
  событие из вот того trace», без искажения формы реального пайплайна ради красивого дерева.
  Вставка в ClickHouse — дочерний span `clickhouse.insert_batch` того же батчевого span'а.
- **`pulse-api`:** автоинструментация FastAPI (`opentelemetry-instrumentation-fastapi`) создаёт
  span на каждый HTTP-запрос. `TraceIdMiddleware` (ADR 0006) вместо случайного `uuid4()` берёт
  `trace_id` активного OTel-span'а — тот же самый идентификатор, что виден в Jaeger, попадает в
  `X-Trace-Id` и в поле `trace_id` структурных логов без изменения существующего контракта.

## Явно НЕ решается этим ADR

**«Один trace_id от приёма события до его появления в ответе API»** буквально, как один and тот же
span-граф, не реализуется — и не должен: ответ `/api/v1/trending` агрегирует произвольное число
событий за окно, у него не может быть один causal predecessor. Демонстрируется другое: (1) ingest
пайплайн — от `kafka.produce` в `gh-collector` до `clickhouse.insert_batch` в `pulse-consumer` —
виден в Jaeger как один связный граф трейсов через links; (2) API-запрос — от HTTP-входа до ответа
клиенту — виден как одно дерево span'ов с тем же `trace_id`, что в логе и в `X-Trace-Id`. Оба
пайплайна инструментированы одним и тем же SDK, экспортируются в один Jaeger и видны на одной
карте сервисов — этого достаточно, чтобы найти узкое место на любой стадии, не обещая связность
там, где её структурно не может быть.

## Отвергнутые альтернативы

- **Собственный OTel Collector перед Jaeger:** оправдан, когда нужен fan-out в несколько backend'ов
  или сэмплирование на отдельном узле. Здесь один backend (Jaeger) и он сам принимает OTLP —
  лишний прыжок сети и лишний контейнер без пользы.
- **Один span на всё сообщение батча (parent-child вместо links):** потребовал бы либо вставлять
  события в ClickHouse по одному (противоречит styleguide §3.3 и критерию приёмки задачи 1.6.4 —
  колоночный батч, не построчный INSERT), либо присваивать span'у вставки один случайно выбранный
  родительский trace, что было бы честнее назвать багом наблюдаемости, чем трассировкой.
- **Трассировка каждого события в backfill-режиме без сэмплирования:** при разборе исторического
  дампа GH Archive (потенциально 100M+ событий) span на каждое означал бы генерацию сопоставимого
  числа span'ов ради разового прогона — Jaeger не рассчитан держать такой объём ради дампа, который
  и так уже лежит в ClickHouse. Решение — стандартный `OTEL_TRACES_SAMPLER=traceidratio` (переменная
  окружения OTel SDK, без своего кода): по умолчанию (dev, `--live`) сэмплирование не включено
  (`parentbased_always_on`), для холодного backfill — понизить через переменную окружения перед
  запуском.

## Последствия

- Новый инфраструктурный контейнер (`jaeger`) в `docker-compose.yml` — не рантайм-сервис продукта
  (см. ограничение «ровно два рантайм-сервиса плюс консьюмер» в CLAUDE.md), тот же класс, что уже
  занимают `prometheus`/`grafana`.
- W3C `traceparent` в заголовках Kafka-сообщения — часть контракта топика `gh.events` наравне с
  ключом и значением; новый консьюмер этого топика, если он не читает заголовки, просто не увидит
  links — ни новых событий, ни отказа это не вызывает.
- Сэмплирование по умолчанию отключено (`always_on`) — план 3.4 (нагрузочный k6-тест) обязан
  учитывать, что каждый запрос в API и каждое событие в `--live` производят span, прежде чем снимать
  цифры RPS/latency под нагрузкой; иначе накладные расходы трассировки исказят замер.
