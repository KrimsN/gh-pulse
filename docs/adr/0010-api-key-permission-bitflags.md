# ADR 0010: Битовые флаги для уровня доступа API-ключа к `/admin`

**Статус:** принято
**Дата:** 2026-07-21

## Контекст

Выпуск API-ключа (задача 2.5) был доступен только через CLI (`app.cli create-key`) — на клоне без
docker exec-доступа к контейнеру `pulse-api` это лишний барьер. Панель `/admin` (задача 4.4) уже
даёт HTTP Basic Auth поверх того же секрета, но пока целиком read-only — выпустить ключ из неё
нельзя. Задача 4.5 добавляет выпуск ключей прямо в `/admin`, и вместе с этим — три уровня доступа к
самой панели: полный доступ (включая выпуск новых ключей), доступ только к существующим read-only
разделам (наполненность данных, генератор бэкфила, логи), и полное отсутствие доступа к `/admin`
(ключ работает только как `X-API-Key` для `/api/v1/*`, как и раньше).

## Решение

Доступ ключа к `/admin` хранится одним целым числом — `api_keys.permissions SMALLINT`, битовые
флаги (`ApiKeyPermission(IntFlag)` в `app/security/keys.py`):

```python
class ApiKeyPermission(IntFlag):
    NONE = 0
    ADMIN_READ = 1 << 0   # /admin: наполненность, бэкфил-команда, логи, дашборд
    ADMIN_WRITE = 1 << 1  # /admin/keys: список и выпуск ключей
```

Три уровня из требования — не отдельные допустимые значения колонки, а именованные пресеты поверх
битов (`ROLE_PRESETS` в том же модуле): `admin = ADMIN_READ | ADMIN_WRITE`, `maintenance =
ADMIN_READ`, `api_only = NONE`. CLI (`--role`) и форма `/admin/keys` работают с этими тремя именами;
в БД и во всех проверках доступа — только целое число и побитовое `&`.

Доступ **не влияет** на `X-API-Key`/`/api/v1/*` — там продолжает действовать одна проверка «ключ
активен», без учёта `permissions`; `app/security/api_key.py` не читает эту колонку вовсе.

`api_only` (`permissions = 0`) на любом `/admin/*` отвечает **401**, не 403 — `require_admin_auth` не
различает «ключа нет», «ключ невалиден» и «ключ валиден, но бит `ADMIN_READ` не установлен», чтобы
анонимный наблюдатель без валидного пароля не получил оракул «этот ключ существует и активен, просто
не та роль». 403 используется только между `maintenance` и `admin` — там наблюдатель уже прошёл
базовый гейт и знает свои собственные биты, различие не даёт ему новой информации.

`POST /admin/keys` — первый роут `/admin` с реальным побочным эффектом (до этой задачи даже
`/admin/backfill-command` только форматирует команду для копирования, ничего не пишет). HTTP Basic
Auth уязвим к CSRF иначе, чем cookie-сессии: браузер сам подставляет закэшированный `Authorization`
на кросс-origin POST, инициированный сторонней страницей в той же сети, пока вкладка `/admin`
открыта — `SameSite` тут не защищает. Вместо полноценных сессий/CSRF-токенов — точечная сверка
`Origin`/`Referer` с собственным host только на этом одном роуте.

## Отвергнутые альтернативы

- **`TEXT` + `CHECK (role IN ('admin', 'maintenance', 'api_only'))`.** Первоначальный план для этой
  задачи. Каждый новый уровень доступа в будущем требовал бы миграции, переписывающей список
  допустимых строк в `CHECK`. Битовые флаги добавляют новый уровень одной строкой Python-константы,
  без миграции, пока значение помещается в `SMALLINT`.
- **Postgres native `CREATE TYPE ... AS ENUM`.** В проекте нет прецедента: весь runtime-доступ к
  Postgres — сырой SQL через `asyncpg`, ORM-слоя и типизированных объектов схемы в БД принципиально
  нет (см. докстроку миграции `0001`). Заводить native enum ради одной колонки — новый паттерн без
  выигрыша, который не даёт `IntFlag` на стороне Python.
- **Полноценные сессии/CSRF-токены для `/admin`.** Решило бы CSRF строже, но требует ровно того
  механизма аутентификации (сессии, cookie), который проект осознанно не заводит — см. ADR 0005 и
  «Осознанно не делаем» в `docs/ARCHITECTURE.md`. Точечная проверка `Origin`/`Referer` закрывает
  конкретный новый риск (первый write-роут) без новой инфраструктуры.

## Последствия

- Расширение прав в будущем (например, отдельный бит на запуск бэкфила из `/admin`, а не только
  генерацию команды) — новая константа `ApiKeyPermission`, без миграции схемы, пока используется
  меньше 15 бит (`SMALLINT`).
- `docs/ARCHITECTURE.md` не дублирует расшифровку битов и это рассуждение — ссылается на
  `ApiKeyPermission` в `app/security/keys.py` и на этот ADR.
- UI/CLI намеренно открывают только три именованных пресета, не редактирование отдельных битов —
  расширять интерфейс до гранулярных прав можно позже, независимо от модели хранения.
