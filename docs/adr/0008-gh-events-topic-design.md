# ADR 0008: Дизайн топика gh.events — ключ event_id, шесть партиций, retention по времени

**Статус:** принято
**Дата:** 2026-07-16

## Контекст

Между Go-коллектором и Python-консьюмером стоит Redpanda (ADR 0003), но форма топика до сих пор не
зафиксирована. Решить нужно три вещи, и каждая плохо поддаётся отмене: ключ сообщения (определяет
равномерность нагрузки), число партиций (уменьшить его нельзя) и retention (умеет терять данные
молча).

Ключ здесь выбирается свободнее, чем кажется. Дедупликация уже решена на слое данных: события несут
`event_id`, повторы безопасны (ADR 0004), и ключ Kafka для этого не нужен. Порядок доставки тоже
ничего не диктует — ClickHouse хранит события в `ORDER BY (event_type, created_at, repo_id)` и к
порядку вставки безразличен. Значит ключ отвечает ровно за одно: как события лягут по партициям.

Ходовое опасение — партиционировать по `repo_id` нельзя, потому что гиганты вроде `torvalds/linux`
создадут горячую партицию. Опасение выглядит правдоподобно, но это утверждение о данных, а такие
утверждения в этом репозитории проверяются, а не пересказываются.

### Измерение

Два реальных часа GH Archive, по одному из каждой эпохи данных (ADR 0007): `2024-06-03-15` — 256 682
события, та эпоха, из которой идёт бэкфилл; `2026-06-01-15` — 149 565 событий, живой поток.

Сначала то, что не зависит от хэш-функции — частоты ключей:

| | 2024-06-03-15 | 2026-06-01-15 |
|---|---|---|
| Уникальных `event_id` | 256 682 (= числу событий) | 149 565 (= числу событий) |
| Уникальных `repo_id` | 94 837 | 77 019 |
| Уникальных `actor_id` | 70 482 | 65 967 |
| Крупнейший репозиторий | 4 902 события — **1.91%** часа | 639 — **0.43%** |
| Крупнейший актор | 33 923 — **13.22%** часа | 8 993 — **6.01%** |
| Крупнейший тип события | `PushEvent` — **53.74%** | `PushEvent` — **84.75%** |

Крупнейший репозиторий часа занимает 1.91% — и это не `torvalds/linux`, а бот-репозиторий
`AIbotsolutions/ERPBOTBOT`. При шести партициях такой ключ добавляет своей партиции 0.11 среднего:
горячей партиции из него не выходит. Зато перекос обнаружился там, где его не искали: один актор
(`github-actions[bot]`) даёт 13.22% часа.

Затем сквозной прогон: ключи каждого часа залиты в топики по шесть партиций, распределение снято по
high watermark. Перекос = максимум партиции к среднему.

| Ключ | Перекос 2024 | Перекос 2026 | Распределение по партициям (2024) |
|---|---|---|---|
| `event_id` | **1.005** | **1.012** | 42873 42785 42597 42999 42636 42792 |
| `repo_id` | **1.142** | **1.047** | 48851 43720 41068 40545 42065 40433 |
| `actor_id` | 1.717 | 1.299 | 36006 46532 33808 32705 34158 73473 |
| `event_type` | 3.458 | 5.093 | 0 25199 45749 36801 991 147942 |
| константа (контроль) | 6.000 | 6.000 | 0 0 0 256682 0 0 |

Два контроля подтверждают, что метод измеряет то, что заявлено. Константный ключ обязан лечь в одну
партицию — лёг (позитивный контроль: ключи действительно участвуют в партиционировании). `event_type`
обязан дать явный перекос — дал (негативный контроль: метод способен увидеть горячую партицию).
Без них «всё равномерно» было бы неотличимо от сломанного измерения.

Читается это так. Опасение про горячие партиции по `repo_id` **не подтвердилось**: 1.14 и 1.05.
Но у измерения есть второй слой, более важный: перекос ключей, зависящих от данных, **дрейфует между
эпохами** — `actor_id` 1.717 → 1.299, `event_type` 3.46 → 5.09, `repo_id` 1.14 → 1.05. У `event_id`
он стоит на месте: 1.005 → 1.012. Равномерность `event_id` — свойство самого ключа (все значения
уникальны), а равномерность `repo_id` — свойство сегодняшних данных.

Воспроизведение (любой час, любой кандидат):

```bash
curl -fsSL https://data.gharchive.org/2024-06-03-15.json.gz -o hour.json.gz
python - <<'PY' > repo-id.tsv
import gzip, json
with gzip.open("hour.json.gz", "rt", encoding="utf-8") as fh:
    for line in fh:
        e = json.loads(line)
        print(f'{e["repo"]["id"]}\t{e["id"]}')
PY
rpk topic create skew-repo-id -p 6 -r 1
rpk topic produce skew-repo-id -f '%k\t%v\n' < repo-id.tsv
rpk topic describe skew-repo-id -p     # колонка HIGH-WATERMARK
```

Числа выше получены партиционером `rpk` v24.2.4. Для `event_id` результат от выбора хэша не зависит
вовсе — ключи уникальны, и любая приличная хэш-функция размажет их ровно; для `repo_id` и `actor_id`
на другом клиенте цифры могут отличаться в деталях.

## Решение

**Ключ сообщения — `event_id`. Шесть партиций у `gh.events`, одна у `gh.events.dlq`. Retention
ограничивается временем, а не размером.**

Ключ `event_id` выбран не потому, что `repo_id` перекошен — измерение показало, что не перекошен, —
а по трём причинам:

- Его равномерность структурная. `event_id` уникален по определению, и это не изменится от того, что
  завтра какой-нибудь бот начнёт коммитить в один репозиторий втрое активнее.
- Co-partitioning не нужен ни одному потребителю. Дедуп живёт на слое данных (ADR 0004), агрегаты по
  репозиторию считает ClickHouse, а не поток. Локальность по репозиторию не покупает ничего.
- **Шесть партиций — это догадка.** Консьюмера ещё нет, и его реальная параллельность неизвестна;
  цифра взята как стартовая цель. С ключом `event_id` догадку дёшево отменить — партиции можно
  добавить, и семантика не пострадает. С `repo_id` добавление партиций перемешало бы соответствие
  «репозиторий → партиция», сломав ровно то co-partitioning, ради которого такой ключ и берут.

Конфигурация топиков:

| Свойство | `gh.events` | `gh.events.dlq` |
|---|---|---|
| Партиции | 6 | 1 |
| Репликация | 1 | 1 |
| `cleanup.policy` | `delete` | `delete` |
| `retention.ms` | 86 400 000 (24 ч) | 604 800 000 (7 суток) |
| `retention.bytes` | не задан | не задан |
| `compression.type` | `zstd` | `zstd` |
| `message.timestamp.type` | `LogAppendTime` | `LogAppendTime` |

**Retention по времени, без лимита по размеру.** Из двух механизмов удаления только лимит по размеру
способен стереть непрочитанное при живом и здоровом консьюмере — просто потому, что продюсер его
обогнал. Это тихое нарушение at-least-once (ADR 0004), замаскированное под настройку диска. Без него
режим отказа обгоняющего продюсера — заполнение диска и громкая остановка приёма записи. Встал
бэкфилл — видно сразу; пропали события — не видно никогда.

Граница безопасности формулируется одной фразой: **максимально допустимый простой консьюмера равен
`retention.ms`**. Сутки покрывают закрытый на ночь ноутбук. Последний рубеж при этом лежит вне
системы: GH Archive неизменяем и качается повторно, поэтому потерянный час бэкфилла восстановим.

**`message.timestamp.type=LogAppendTime`** — метку ставит брокер. С `CreateTime` продюсер проставил
бы время самого события, и час 2024 года оказался бы старше суточного retention **уже в момент
записи**: бэкфилл исчезал бы при первом же роллинге сегмента, молча и целиком. Ровно та ловушка, из-за
которой в таблице ClickHouse нет TTL (`infra/clickhouse/migrations/001_events.sql`).

**Топики создаются одноразовым контейнером `redpanda-init`** (`infra/redpanda/create-topics.sh`) при
каждом `docker compose up`. У образа Redpanda нет хука `/docker-entrypoint-initdb.d`, которым
пользуются ClickHouse и PostgreSQL, поэтому его роль играет отдельный сервис. Скрипт декларативен:
он приводит топик к нужной форме и падает с внятным сообщением, если существующий топик имеет другое
число партиций.

## Отвергнутые альтернативы

- **`repo_id`** — перекос измерен и приемлем (1.14 / 1.05), так что отвергнут не из-за него, а из-за
  цены: он связывает руки по числу партиций ради локальности, которой никто не пользуется.
- **`actor_id`** — перекос 1.717 на часе 2024: один бот (`github-actions[bot]`) даёт 13% событий и
  почти удваивает свою партицию. Локальность по актору к тому же никому не нужна.
- **`event_type`** — 3.46 и 5.09; 15 значений на весь поток, `PushEvent` — большинство. Приведён не
  как кандидат, а как измеренный образец того, как выглядит горячая партиция.
- **Без ключа** (round-robin / sticky-batching) — дал бы равномерность даром и сэкономил байты на
  сообщение, но раскладка стала бы функцией тайминга батчей клиента, а не свойством данных: её
  нельзя ни воспроизвести, ни измерить. Детерминированная раскладка стоит своих байтов: повторный
  прогон того же часа кладёт событие в ту же партицию.
- **Автосоздание топиков** — у брокера включено по умолчанию (`auto_create_topics_enabled=true`), и
  `default_topic_partitions=1`. То есть продюсер, стартовавший раньше инициализации, молча получил бы
  `gh.events` с одной партицией вместо шести: пайплайн работает, параллельность консьюмера тихо
  схлопнута. Опечатка в имени топика по той же причине создавала бы топик-призрак вместо ошибки.
- **Ручной `rpk topic create` в README** — ломает обещание, на котором держится репозиторий: чистый
  клон, одна команда, ноль ручных шагов.
- **Создание топиков из кода сервисов при старте** — админ-логику пришлось бы дублировать на Go и на
  Python, продюсер с консьюмером гонялись бы за право создать топик, а форма топика перестала бы быть
  видна в инфраструктуре.
- **`retention.bytes` как ограничитель** — см. выше: единственный механизм, способный молча удалить
  непрочитанное при живом консьюмере.
- **`cleanup.policy=compact`** — при уникальном ключе компакция не удаляет никогда ничего, то есть
  означает «хранить вечно». Задан `delete` явно, чтобы это не выглядело умолчанием.
- **`CreateTime`** — стёр бы бэкфилл прошлых эпох прямо на вставке.

## Последствия

- События одного репозитория лежат в разных партициях, порядок по репозиторию не гарантирован. Это
  никому не мешает сегодня; если завтра понадобится потоковая агрегация по репозиторию, ключ придётся
  пересматривать — и это будет новый ADR, а не правка этого.
- Число партиций можно наращивать без разрушения семантики. Это прямое следствие выбора ключа и
  главная причина, по которой догадка «шесть» приемлема.
- **Задача 1.4 обязана зафиксировать партиционер явно.** Измерение сделано клиентом `rpk`; если
  Go-клиент возьмёт другую хэш-функцию, цифры для `repo_id`/`actor_id` перестанут описывать систему.
  Для `event_id` риск минимален, но выбор всё равно должен быть записан, а не унаследован от
  умолчания библиотеки.
- **Задача 1.6 обязана падать громко при `OFFSET_OUT_OF_RANGE`.** Retention по времени — гарантия
  только в этом случае: с `auto_offset_reset=earliest` пропавший из-за просрочки диапазон
  перескакивался бы молча, и потеря снова стала бы невидимой.
- Топик держит на диске всё, что произведено за сутки, независимо от того, вычитано это или нет.
  Бэкфилл до 100M событий поэтому идёт диапазонами, а не одним залпом.
- Реальный размер события на диске здесь **не измерен** — продюсера ещё нет. Объём топика и
  коэффициент сжатия станут известны на первом бэкфилле (задача 1.7).
- Перекос измерен при шести партициях; на другое их число эти цифры не переносятся.
