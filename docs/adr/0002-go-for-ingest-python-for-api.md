# ADR 0002: Go для ingest, Python для консьюмера и API

**Статус:** принято
**Дата:** 2026-07-15

## Контекст

У пайплайна две разные по форме нагрузки: высокочастотный I/O-bound цикл fetch/produce и
аналитическое API вместе с батчевым консьюмером. Полиглотность в проекте такого размера нужно
оправдывать: разделение по языкам обязано опираться на инженерные основания, а не на желание
использовать два языка.

## Решение

- **Go для `gh-collector`** (ingest): скачивание множества часов архива и поллинг Events API —
  это конкурентная I/O-bound задача. Горутины и каналы дают дешёвую ограниченную конкурентность и
  небольшой футпринт по памяти при стриминге gzip-файлов на 100–300 МБ. Для Go это идиоматичная работа.
- **Python для `pulse-consumer` и `pulse-api`**: батч-вставка консьюмера и аналитический SQL API —
  это работа с async I/O и сложным SQL, где экосистема Python сильна. `clickhouse-connect` и
  `asyncpg` зрелые; FastAPI даёт async-стек, валидацию через Pydantic и OpenAPI из коробки.

Граница (Kafka) — естественный шов: Go продюсит, Python консьюмит. Каждый язык делает то, в чём он
лучше, и выбор объясняется одним предложением.

## Отвергнутые альтернативы

- **Всё на Python:** работало бы. Ingest на asyncio справился бы с текущими объёмами, но горутины
  дают более дешёвую ограниченную конкурентность и меньший футпринт по памяти на стриминге крупных
  gzip-файлов — а именно ingest первым упрётся в потолок при росте.
- **Всё на Go:** аналитическое API и SQL-тяжёлый консьюмер дольше делать хорошо на Go, а выигрыша
  на этих нагрузках он не даёт. Незачем воевать с этим.

## Последствия

- Два тулчейна в CI. Приемлемо, оба дёшево гоняются.
- Kafka несёт нагрузку как граница между языками — это усиливает выбор настоящего брокера (ADR 0003).
