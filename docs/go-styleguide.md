# Go — style guide

Этот документ описывает договорённости, которые не проверяет линтер: как раскладывать код по
пакетам, как оборачивать и различать ошибки, как использовать `context.Context` и каналы, как
писать потоковую обработку данных и тесты. Механические правила (форматирование, `go vet`,
неиспользуемые переменные, незакрытые HTTP-тела, сравнение ошибок через `==` вместо `errors.Is`)
заданы в [`.golangci.yml`](../services/gh-collector/.golangci.yml) — это источник истины, здесь они
не дублируются. Применяется к `gh-collector` — единственному и постоянному Go-сервису проекта
(см. «Ограничения» в архитектуре: второй Go-сервис не заводится).

## 1. Именование и структура кода

### 1.1 Раскладка сервиса

Стандартная Go-раскладка: `cmd/` — точка входа, `internal/` — всё остальное. `pkg/` не заводится,
пока у модуля нет внешних потребителей за пределами этого репозитория:

```
services/gh-collector/
  cmd/gh-collector/
    main.go        # CLI: разбор флагов, сигналы, маппинг ошибок в exit code
  internal/
    archive/        # скачивание и потоковый разбор дампов GH Archive
    model/           # каноническая схема события + нормализация из проволочного формата
```

Пакет — на одну ответственность, а не на один файл: `archive` не знает про формат CLI-флагов,
`model` не знает про HTTP. Граница проходит там же, где и в Python-сервисах — раздел 1.1
[docs/python-styleguide.md](python-styleguide.md).

### 1.2 Именование

- Экспортируемые имена — `PascalCase`, неэкспортируемые — `camelCase`. Тип, который существует
  только для формы проводов, а не как публичный API пакета, — неэкспортируемый (пример:
  `githubEvent`, `eventRef` в `internal/model/event.go` — наружу торчит только канонический
  `Event`).
- Имя метода не повторяет имя получателя (`archive.FetchHour`, а не
  `archive.ArchiveFetchHour`) и не тащит контекст, который уже понятен из имени пакета.
- Имена получателей (receiver) — короткие, одна-две буквы, последовательные в пределах типа.
- Sentinel-ошибки — `errXxx` (неэкспортируемая) или `ErrXxx` (экспортируемая, когда вызывающему
  коду вне пакета нужно сверяться через `errors.Is`) — см. 2.3.

### 1.3 Пакетные глобальные переменные

`baseURL` и `httpClient` в `internal/archive/archive.go` — неэкспортируемые пакетные переменные,
подменяемые в тестах на `httptest.Server` и клиент с другими таймаутами. Это осознанное отступление
от подхода «внедрять зависимость через поле структуры» (который правильнее для библиотеки с
несколькими потребителями): пакет сейчас — один файл с одной точкой входа (`FetchHour`), и
struct-обвязка добавила бы косвенность без выигрыша. Префикс `_` перед именем таких переменных
(конвенция части сообщества для явного отличия пакетного глобала от локальной переменной) здесь не
применяется по той же причине — пакет из одного файла, спутать не с чем.

Пересмотреть оба решения стоит, как только у пакета появится вторая точка входа или тест, которому
нужен собственный независимый клиент, — тогда разница между «одна пара глобалов на пакет» и
«клиент как поле структуры» перестанет быть чисто гипотетической.

### 1.4 Комментарии и doc-comments

Каждый файл начинается с doc-comment на пакет (`// Package archive ...`). Каждое экспортируемое имя
документируется отдельным комментарием — полным предложением, начинающимся с этого имени:

```go
// FetchHour стримит события одного часа GH Archive в out. Останавливается по ctx: как только
// контекст отменён (например, SIGINT из cmd/gh-collector), функция прекращает чтение на текущей
// строке и возвращает ctx.Err() ...
func FetchHour(ctx context.Context, hour time.Time, out chan<- model.Event) error
```

Комментарии — по-русски (общее правило проекта) и объясняют **почему**, а не **что** — код уже
показывает что. Образцовый случай — `parseHour` в `cmd/gh-collector/main.go`: комментарий фиксирует
не алгоритм разбора (он и так виден), а причину, по которой `time.Parse` выбран вместо
`fmt.Sscanf` — с конкретными наблюдёнными различиями в поведении на некорректном вводе. Это тот
уровень детализации, который стоит комментария: не то, что делает функция, а то, какую альтернативу
она сознательно отвергла и почему.

## 2. Ошибки

### 2.1 error — значение, не исключение

`panic` не используется для ожидаемых сбоев (сеть недоступна, битый JSON, неверный флаг) — только
`error` как возвращаемое значение. `panic`/`recover` оставлены для действительно невосстановимых
программных ошибок (нарушенный инвариант), которых в текущем коде нет.

### 2.2 Обёртывание

`fmt.Errorf("...: %w", err)` — на каждой границе, где ошибка поднимается на уровень выше и туда
стоит добавить контекст (какой файл, какой час, какой URL). Когда вызывающему коду нужно различить
и sentinel-ошибку, и исходную причину, — двойной `%w` (Go 1.20+):

```go
// %w дважды, а не %w + %v: errUsage нужен main для кода возврата 2, но причина от parseHour
// должна остаться доступной errors.Is — с %v она превратилась бы в текст и перестала быть ошибкой.
return fmt.Errorf("%w: разбор --hour: %w", errUsage, err)
```

### 2.3 Sentinel-ошибки

`errors.New` на уровне пакета, сравнение — только через `errors.Is`, никогда через `==`
(это уже ловит `errorlint` из `.golangci.yml`, но сама причина важна и для читателя): текст ошибки
может обрасти обёрткой на пути вверх, и только `errors.Is` умеет пройти сквозь `%w`-цепочку.

```go
// errUsage — сигнальная ошибка неверных флагов/аргументов. main отличает её от сбоя fetch через
// errors.Is, чтобы вернуть exit code 2 (конвенция CLI-утилит), а не 1.
var errUsage = errors.New("usage")
```

### 2.4 Осознанное игнорирование ошибки

`_ = expr` допустим только с комментарием, объясняющим, почему с этой конкретной ошибкой на этом
конкретном месте действительно нечего делать:

```go
// Ошибку закрытия глушим осознанно, а не по недосмотру: тело только читается, и его Close
// сообщает лишь о сбое возврата соединения в пул — сделать с этим на выходе из функции нечего,
// а подменять ею настоящую ошибку чтения (её возвращаем ниже) было бы прямым вредом.
defer func() { _ = resp.Body.Close() }()
```

`_ = expr` без комментария не проходит ревью — это неотличимо от забытой проверки, а
`errcheck`-нарушение, которое молча подавили, опаснее того, которое просто ловит линтер.

### 2.5 Штатная остановка — не ошибка выполнения

Отмену через `context.Canceled` различаем от настоящего сбоя явным `errors.Is`, а не общим
`err != nil`:

```go
case errors.Is(err, context.Canceled):
    fmt.Fprintln(os.Stderr, "остановлено пользователем (SIGINT), час докачан не полностью")
    os.Exit(130) // конвенция shell: 128 + SIGINT(2)
```

## 3. Конкурентность, context и потоковая обработка

### 3.1 context.Context — первый параметр везде, где есть I/O или отмена

Любая функция, которая делает сетевой запрос, читает поток или пишет в канал, принимает
`context.Context` первым параметром. Сигналы ОС заводят корневой контекст один раз, на входе в
программу — `signal.NotifyContext(context.Background(), os.Interrupt)` в `main.go`, не глубже.

### 3.2 Размер буфера канала — обоснованное число, не круглое на глазок

```go
// Буферизованный канал: FetchHour может уйти вперёд печати сэмпла на несколько сотен событий,
// не блокируясь после каждой строки. В задаче 1.4 этот же канал станет входом воркер-пула
// Kafka-продюсера — размер буфера здесь уже задаёт форму backpressure между fetch и produce.
events := make(chan model.Event, 1000)
```

Backpressure — осознанный выбор архитектуры, а не побочный эффект: канал без буфера или с
буфером «про запас» — оба одинаково скрывают, кто на самом деле кого тормозит.

### 3.3 Контракт владения каналом — явно в doc-comment

Кто закрывает канал — часть публичного контракта функции, а не то, что читатель должен
реконструировать по вызывающему коду:

```go
// FetchHour не закрывает out — это осознанное отступление от идиомы «отправитель закрывает
// канал»: закрытие остаётся на вызывающем коде. В задаче 1.4 несколько вызовов FetchHour (по
// одному на час бэкфилла) будут писать в один общий канал воркер-пула продюсера, и закрыть его
// сможет только тот, кто знает, что закончили все отправители разом ...
func FetchHour(ctx context.Context, hour time.Time, out chan<- model.Event) error
```

### 3.4 У каждой горутины — предсказуемая точка останова

Горутина, у которой нет способа сигнализировать остановку и нет способа дождаться её завершения, —
утечка. В текущем коде это решено закрытием канала как happens-before сигналом (без лишнего
`sync.WaitGroup` там, где закрытие канала уже даёт нужную гарантию):

```go
var fetchErr error
go func() {
    fetchErr = archive.FetchHour(ctx, hour, events)
    close(events) // сигнал консьюмеру ниже: событий больше не будет
}()

for evt := range events {
    // ...
}
// Закрытие events happens-before выход из range по гарантии рантайма для чтения из закрытого
// канала, а close(events) в горутине выше идёт строго после присваивания fetchErr — поэтому
// здесь fetchErr уже виден без дополнительной синхронизации.
```

Когда в коде появится воркер-пул с несколькими одновременными горутинами (задача 1.4), для тестов
пакетов, которые их спавнят, — [`go.uber.org/goleak`](https://pkg.go.dev/go.uber.org/goleak):
единичная горутина с ручной happens-before цепочкой достаточно проста для ручного рассуждения,
пул — уже нет.

### 3.5 HTTP-клиент — свой, не `http.DefaultClient`

`http.DefaultClient` — общий изменяемый пакетный глобал: любой другой код в процессе может
подменить его или рассчитывать на другие таймауты. Свой клиент — с таймаутами по фазам соединения,
а не одним общим `Timeout`, когда тело ответа читается потоково и может занимать минуты:

```go
// У клиента намеренно нет общего Timeout: он покрывает всё время запроса вместе с телом ответа,
// а час 2024 года — это больше гигабайта, который легально качается несколько минут даже на
// быстрой сети. Вместо этого ограничены только фазы установления соединения и заголовков.
var httpClient = &http.Client{
    Transport: &http.Transport{
        DialContext:           (&net.Dialer{Timeout: 10 * time.Second}).DialContext,
        TLSHandshakeTimeout:   10 * time.Second,
        ResponseHeaderTimeout: 30 * time.Second,
        IdleConnTimeout:       90 * time.Second,
    },
}
```

### 3.6 Потоковая обработка вместо загрузки в память целиком

Час GH Archive — от ~100 МБ до ~1.1 ГБ в распакованном виде (см.
[ADR 0007](adr/0007-hybrid-data-epochs.md)). `io.ReadAll` на теле такого ответа — не вариант; поток
читается построчно через `gzip.Reader` + `bufio.Scanner`, ни разу не собираясь в памяти целиком.

`bufio.Scanner`, а не `json.Decoder` напрямую поверх потока: после ошибки `Decode()` позиция в
потоке произвольная, и продолжать декодировать следующие объекты небезопасно. Формат GH Archive —
JSON Lines (один объект на строку), поэтому `Scanner` даёт естественную точку ресинхронизации —
начало следующей строки, и одна битая строка не выбивает из потока весь оставшийся час. Верхний
предел длины строки задан явно и трактуется как разные по серьёзности случаи: одна битая, но не
длиннее предела строка — логируется и пропускается; строка длиннее предела — фатальный сбой всего
часа (при типичном размере события в единицы-десятки КБ такая длина означает порчу потока, а не
большой payload).

## 4. Тестирование и логирование

### 4.1 Только stdlib `testing`, table-driven

Без testify — сравнение через обычный `if got != want`, табличные тесты с `t.Run` на каждый кейс,
имя кейса описывает *условие*, а не номер:

```go
tests := []struct {
    name    string
    input   string
    want    time.Time
    wantErr bool
}{
    {
        name:    "час 24 — раньше молча переносился на следующий день, час 0",
        input:   "2026-06-01-24",
        wantErr: true,
    },
    // ...
}
for _, tt := range tests {
    t.Run(tt.name, func(t *testing.T) {
        got, err := parseHour(tt.input)
        // ...
    })
}
```

Название кейса объясняет, зачем он вообще существует, — часто это описание конкретного бага,
который тест закрывает, а не только формы входа.

### 4.2 HTTP и потоковый разбор — `httptest` и golden-файлы

Путь через реальную сеть подменяется `httptest.Server` (для этого `baseURL`/`httpClient` в
`archive.go` — пакетные переменные, см. 1.3). Форма реальных данных проверяется golden-файлом
(`internal/archive/testdata/sample_hour.jsonl`) — таблично собранные вручную кейсы не заменяют
проверку на настоящем куске дампа, где встречаются реальные комбинации полей.

### 4.3 Round-trip тесты для симметричных DTO

Там, где тип объявлен симметричным (`json.Marshal` → `json.Unmarshal` даёт то же значение, как
`Event` в `internal/model/event.go`), это фиксируется отдельным тестом на сам факт симметрии, а не
только тестами на разбор в одну сторону — регрессия симметрии иначе не поймана ничем.

### 4.4 Логирование

Для нового кода — `log/slog`: структурные поля вместо интерполяции в текст, чтобы поле можно было
агрегировать, а не парсить из строки:

```go
slog.Info("событие пропущено", "hour", label, "line", lineNum, "err", err)
```

Формально решение зафиксировано только здесь — отдельного ADR под логирование Go-сервиса, в отличие
от [ADR 0006](adr/0006-structlog-for-logging.md) для Python-стороны, пока нет. Единственный
существующий вызов `log.Printf` (`internal/archive/archive.go`) под это правило не подгоняется
задним числом — правило действует для кода, который пишется дальше, включая Kafka-продюсер и
воркер-пул задачи 1.4, где структурные поля (какой час, какой воркер, какой батч) пригодятся сразу.
