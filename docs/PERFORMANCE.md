# Производительность

История оптимизации GH Pulse: что было медленно, почему, что изменилось и сколько это дало.

Документ ведётся с первого дня. Первая запись — baseline `/trending` (задача 1.9, Фаза 1); история
оптимизаций разворачивается дальше в Фазе 3 (см. [TASKS.md](../TASKS.md)). Здесь же зафиксировано
правило, по которому сюда попадает любое число.

## Железное правило

**Ни одна цифра в этом репозитории не пишется руками.** Каждое число — README, этот файл, коммиты —
обязано трассироваться к воспроизводимому прогону бенчмарка из [`bench/`](../bench) и к коммиту,
на котором прогон сделан. Цифра без прогона за спиной — это не измерение, а впечатление.

Отсюда же следует, что здесь не будет круглых «примерно в 50 раз быстрее». Будет «2 340 мс → 45 мс,
rows_read 112.4M → 8 760, датасет 112.4M событий, одноузловой ClickHouse».

Второе правило — **только реальные данные**: замеры идут на настоящих событиях
[GH Archive](https://www.gharchive.org/), синтетика не генерируется. Профиль нагрузки (распределение
типов событий, перекос по репозиториям, суточные пики) — часть измеряемого, и на сгенерированных
данных он был бы выдуманным.

## Формат записи

Каждая оптимизация описывается одним блоком, и все поля обязательны:

| Раздел | Что содержит |
|---|---|
| **Запрос и датасет** | Что меряем, на скольких событиях, на каком железе |
| **Baseline** | Медиана warm (несколько прогонов) и cold, `rows_read`, `bytes_read` |
| **Диагноз** | Что показал `EXPLAIN` — почему было медленно, в терминах физического доступа к данным |
| **Изменение** | Ровно одна изменённая вещь, со ссылкой на миграцию или коммит |
| **Результат** | Те же метрики после, плюс подтверждение, что результирующий набор не изменился |
| **Дельта** | Во сколько раз быстрее, насколько меньше прочитано, каким бенчмарком проверено |

Пустой «Диагноз» обесценивает запись: без него выигрыш выглядит случайностью, а не следствием
понимания. Одно изменение на запись — иначе непонятно, что именно дало эффект.

Проверка идентичности результата до и после — не формальность: ускорение, которое незаметно
изменило ответ, это не оптимизация, а баг.

## Trending repos: baseline на прямом скане `events` (без MV)

**Запрос:** `GET /api/v1/trending?window=24h&limit=50` — SQL зафиксирован в
[`bench/trending.sql`](../bench/trending.sql), byte-в-byte то, что строит `build_trending_query`
(`services/pulse-api/app/queries.py`) при этих параметрах (задача 1.8).

**Датасет:** 14 052 941 событие на коммите `0b2c652`, три партиции: `202406` — 13 158 385 строк
(бэкфилл задачи 1.7), `202606` — 149 565 строк (ручная проверка задачи 1.6), `202607` — 744 991
строка (доливка задачи 1.8, последние ~6 часов). Одноузловой ClickHouse в Docker Desktop, хост
16 ядер / ~15 ГБ RAM выделено движку.

### Baseline

Измерено двумя методами — расхождение между ними само по себе диагностическое (см. «Заметки»):

- **End-to-end HTTP** (`curl` → `GET /api/v1/trending`, то есть ровно то, что видит клиент API,
  тот же путь, что и `clickhouse-connect` внутри `pulse-api`): медиана warm (5 прогонов после
  рестарта ClickHouse) **11.0 мс** (разброс 9.8–21.8 мс) · cold (первый запрос сразу после
  рестарта контейнера ClickHouse): **13.6 мс**.
- **Серверная часть запроса** (`system.query_log.query_duration_ms` для тех же запросов,
  HTTP-интерфейс): **4–17 мс**, медиана ≈ 5 мс — подтверждает, что почти всё время в HTTP-цифре
  выше это сама ClickHouse-обработка, а не сериализация или сеть.
- rows_read: **65 055** · bytes_read: **2 983 519** (≈2.85 MiB несжатых по `system.query_log`),
  **1 659 794** (≈1.58 MiB) реально прочитано с диска (`ReadCompressedBytes`).

### Диагноз

```
EXPLAIN indexes = 1 <bench/trending.sql>

ReadFromMergeTree (ghpulse.events)
Indexes:
  MinMax
    Keys: created_at
    Condition: (created_at in [1784184122, +Inf))
    Parts: 8/22       Granules: 91/1722
  Partition
    Keys: toYYYYMM(created_at)
    Condition: (toYYYYMM(created_at) in [202607, +Inf))
    Parts: 8/8         Granules: 91/91
  PrimaryKey
    Keys: event_type, created_at
    Condition: and((created_at in [1784184122, +Inf)), (event_type in ['WatchEvent', 'WatchEvent']))
    Parts: 8/8         Granules: 8/91
```

Этот «наивный» запрос — **не** полный скан таблицы. `PARTITION BY toYYYYMM(created_at)` и
`MinMax`-индекс по `created_at` целиком исключают партиции `202406` и `202606` (13.3M событий не
читается ни байтом), а `PrimaryKey`-индекс `(event_type, created_at, repo_id)` — выбор задачи 1.2 —
дальше сужает оставшуюся партицию до 8 из 91 гранулы. Итог: 65 055 прочитанных строк из
14 052 941 (0.46%), и запрос укладывается в единицы миллисекунд на серверной стороне. Хорошая
эргономика `ORDER BY` уже окупается здесь без всякого MV.

Настоящая, неисправленная этим baseline'ом проблема — не «скан», а **отсутствие
предагрегации**: каждый запрос заново читает и суммирует 65 055 сырых `WatchEvent`, хотя ответ —
не больше 50 строк. `rows_read` растёт с объёмом событий *внутри окна*, а не с размером ответа;
на нынешнем датасете окно короткое и датасет скромный, поэтому это дёшево. При непрерывном live-
поллинге (задача 2.9) и росте до 100M+ (Фаза 3) тот же запрос будет читать пропорционально больше
сырых строк на каждый запрос — ровно то, что почасовой `SummingMergeTree` MV (задача 2.1/2.3)
исправляет: его стоимость чтения ограничена числом (репозиторий × час) в окне, а не темпом ingest.

### Заметки

- **CLI-обёртка (`clickhouse-client` через `docker compose exec`) как метод измерения отброшена.**
  Первая серия прогонов таким способом показала 440–700 мс — это оказался почти целиком шум спавна
  процесса и TCP-хендшейка `docker exec`, а не стоимость запроса: те же самые запросы в
  `system.query_log` показывают `query_duration_ms` 4–19 мс. Тот же принцип протокола («rows_read
  не упал, а время упало → значит измерили не то») сработал и здесь, только для метода измерения
  целиком, а не для конкретного прогона — поэтому в Baseline пошли HTTP-путь эндпоинта и
  `system.query_log`, а не CLI wall time.
- **`bench_query.py` не запустился напрямую на этой машине** (Windows, Python из `uv`):
  `subprocess.run(["clickhouse-client", ...], shell=False)` на Windows резолвится через нативный
  `CreateProcess`, который ищет исполняемый файл только с расширением `.exe` (в отличие от
  `cmd.exe`, который умеет `PATHEXT`) — ни bash-скрипт без расширения, ни `.cmd`-обёртка над
  `docker compose exec clickhouse clickhouse-client` не находятся этим путём. Цифры этой записи
  сняты эквивалентным протоколом вручную (тот же `bench/trending.sql`, серии прогонов, медиана,
  `rows_read`/`bytes_read` из `--print-profile-events` и `system.query_log`).
  [`bench/trending.sql`](../bench/trending.sql) остаётся источником правды для запроса и
  воспроизводим через `bench_query.py` в любом окружении, где `clickhouse-client` есть нативно на
  `PATH` (Linux/macOS/WSL).
- **Окно `7d` на этом датасете читает те же 65 055 строк, что и `24h`**: партиция `202607`
  целиком — это ~745K событий за последние ~6 часов (доливка задачи 1.8), более старых данных в
  ней нет. Нынешняя скорость — отчасти артефакт скудности «живого» хвоста датасета, а не только
  заслуга схемы; с непрерывным live-поллингом (2.9) `rows_read` для `7d` вырастет на порядки
  относительно `24h`, и разрыв между baseline и MV станет заметнее, чем на этом снимке.
- Дважды перезапускал контейнер ClickHouse между сериями замеров, чтобы снять `cold` без прогретого
  mark cache/uncompressed cache; `rows_read`/`bytes_read` идентичны на каждом прогоне (детерминизм,
  как и обещает протокол измерений).
